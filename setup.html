🛠 Database table + GitHub setup & launch sequence
A · Create the database table one time · Supabase → SQL Editor → Run · safe to re-run
-- WIPES the jobs table and rebuilds it fresh to this EXACT schema.
-- Use this if a previous table was a different shape and caused "no match" / "permission denied".
drop table if exists public.jobs cascade;
create table public.jobs (
  idx             bigint generated always as identity,
  id              text primary key,
  company         text not null,
  title           text not null,
  location_hub    text,
  location        text,
  salary_range    text,
  salary_min      int,
  salary_max      int,
  intern_rate     int,
  intern_weeks    int,
  is_intern       boolean default false,
  type            text,
  match           int,
  planets         int,
  demand          int,
  posting_age     int,
  tech            text,
  clearance       text,
  relocation_flag boolean default false,
  relocation_note text,
  logo_bg         text,
  url             text,
  timestamp       timestamptz,
  description     text,
  requirements    text,
  source          text,
  status          text default 'active',
  saved           boolean default false,
  archived        boolean default false,
  itar_flag       boolean default false,
  clearance_level text,
  degree_levels   text,
  hw_sw_track     text,
  housing_stipend boolean default false,
  intern_skills   text,
  apply_deadline  timestamptz,
  program_start   timestamptz,
  created_at      timestamptz default now()
);
alter table public.jobs enable row level security;
drop policy if exists "anon read jobs" on public.jobs;
create policy "anon read jobs" on public.jobs
  for select to anon using (true);
-- GRANT table permissions — fixes "permission denied for table jobs" even with the
-- correct service_role key (the role needs an explicit grant to write).
grant usage on schema public to anon, authenticated, service_role;
grant select on public.jobs to anon, authenticated;
grant all privileges on public.jobs to service_role;
grant insert, update, delete on public.jobs to service_role;
-- Single-user tool: disable row-level security so the service_role key can always write.
alter table public.jobs disable row level security;
-- Belt & suspenders: an explicit write policy (in case the key resolves as authenticated).
drop policy if exists "service role writes" on public.jobs;
create policy "service role writes" on public.jobs
  for all to service_role using (true) with check (true);
create index if not exists jobs_hub_time_idx
  on public.jobs (location_hub, timestamp desc);
-- Supabase's REST layer (PostgREST) caches role permissions and does NOT always pick up
-- GRANT/RLS changes immediately — this NOTIFY forces an instant cache reload so the fix
-- above takes effect right away instead of waiting for PostgREST's automatic refresh.
NOTIFY pgrst, 'reload schema';
Still getting "permission denied for table jobs" after running the SQL above? That error only ever means one thing at the database level: the role writing to the table is missing a GRANT — it is not caused by Row-Level Security (RLS violations show a different message). If the big script above ran into any error partway through, later lines (including the GRANT) may not have executed. Run just this one isolated block by itself — it can't be blocked by anything earlier failing: 
grant usage on schema public to service_role, anon, authenticated;
grant all privileges on public.jobs to service_role;
grant select on public.jobs to anon, authenticated;
alter table public.jobs disable row level security;
NOTIFY pgrst, 'reload schema';
Paste that alone into Supabase → SQL Editor → Run, confirm "Success. No rows returned", then try Connect & fetch real jobs again. If it STILL fails after this exact isolated block succeeds, the key in field 3 is not actually authenticating as service_role against this project (even if it decodes that way) — regenerate the service_role key from Settings → API and paste the fresh one.
⚠ This wipes any existing jobs table (old rows are deleted) and rebuilds it to the exact format the app expects — use it any time a previous table was a different shape and caused "no match" or write errors. The Test connection and Connect & fetch buttons check your live table against this format and pop an alert if anything is still off — if that happens, copy this SQL and run it again.
B · Turn on the daily auto-scraper GitHub Actions · free · runs itself
  1. Make a free github.com account → New repository (Private), name it orbital.
  2. Upload everything from your downloaded orbital folder (include the hidden .github folder).
  3. Repo → Settings → Secrets and variables → Actions → New repository secret. Add two, names exactly:
    SUPABASE_URL = your Project URL  ·  SUPABASE_SERVICE_ROLE_KEY = your service_role key.
  4. Actions tab → “Orbital daily scrape” → Run workflow. After this it runs every day on its own.
C · Launch sequence — verify it all matches
  1. Run the SQL (A) → press Test connection here. You want “✔ the table format matches”.
  2. Paste all three keys → press Connect & fetch real jobs. Watch 0 → 100%, then the live verify list.
  3. Open the dashboard → press Refresh; the count should match the verify list.
Dev notes. Row id = {company}-{ats_id} so the browser pass and the GitHub scraper de-duplicate into the same rows. location_hub is one of SEATTLE / GREATER_LA / NEW_ZEALAND only. The browser pass covers Greenhouse/Ashby/Amazon (CORS-friendly); Workday/Lever and custom sites populate via the daily Action. Every write is checked against the column list above and auto-skips any column your table lacks (logged), so a partial table still loads.

Turn on the GitHub engine

This is the heart of Orbital — the free robot that pulls Blue Origin, Amazon Kuiper and the big boards browsers can't reach, and refreshes everything daily. The dashboard won't open until this is running.

  1. Make the repo. Go to github.com → free account → green New button → name it orbital → Private → Create repository.
  2. Open the upload page. On your new empty repo, click the link “uploading an existing file” (or the Add file ▾ → Upload files button, top right).
  3. Upload the WHOLE orbital folder's contents. Open the orbital-crm folder you downloaded, select everything inside it, and drag it onto that GitHub page. Drop the files and you should see this list appear:
    📄 index.html📄 setup.html 📄 scraper.py📄 requirements.txt 📄 START-HERE.html📄 manifest.json 📁 assets/📁 .github/ ← must be here!
    ⚠ The .github folder is hidden by default — without it the robot won't run. On Mac press Cmd+Shift+. to show hidden files; on Windows turn on View → Hidden items — then drag it in too.
  4. Save it. Scroll down, click the green Commit changes button.
  5. Add your 2 secret keys. In the repo: Settings → Secrets and variables → Actions → New repository secret. Add two (names EXACTLY):
    SUPABASE_URL = your Project URL · SUPABASE_SERVICE_ROLE_KEY = your service_role key.
  6. Run it. Actions tab → click “Orbital daily scrape” on the left → Run workflow ▾ → Run workflow. Wait ~2 min for the green ✓.
